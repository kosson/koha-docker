# TLS & Security

Two independent TLS layers operate in this stack.

## Layer 1: OpenSearch Cluster TLS (mTLS)

### Scope

Node-to-node communication within the OpenSearch cluster, plus admin client ↔ cluster authentication.

### Certificate Files (18 files)

Located in `OpenSearch-3.6/assets/ssl/`:

| File | Purpose | Bind-mounted to |
|---|---|---|
| `root-ca.pem` | Root CA certificate | All nodes + admin |
| `root-ca-key.pem` | Root CA **private key** | All nodes ⚠️ |
| `admin.pem` | Admin client certificate | All nodes |
| `admin-key.pem` | Admin client private key | All nodes |
| `os01.pem` / `os01-key.pem` | Node 1 TLS cert/key | os01 only |
| `os02.pem` / `os02-key.pem` | Node 2 TLS cert/key | os02 only |
| `os03.pem` / `os03-key.pem` | Node 3 TLS cert/key | os03 only |
| `os04.pem` / `os04-key.pem` | Node 4 TLS cert/key | os04 only |
| `os05.pem` / `os05-key.pem` | Node 5 TLS cert/key | os05 only |
| `dashboards.pem` / `dashboards-key.pem` | Dashboards client cert | Dashboards only |

### Transport

- Nodes connect to each other over HTTPS on the transport port (internal only)
- Admin client authenticates with `admin.pem` + `admin-key.pem`
- Health check: `curl -ks --fail --cert admin.pem --key admin-key.pem https://os01:9200/...`

### Generated

The certificates are generated by a script referenced in README.md:
`assets/opensearch/bin/opensearch_local_certificates_creator.sh`

### ⚠️ CA Private Key Exposure

The root CA private key (`root-ca-key.pem`) is **bind-mounted into every OpenSearch container and Dashboards**. This means:
- If any container is compromised, the CA key is exposed
- The CA key is version-controlled in git (unless gitignored — check TRACKER.md)
- Anyone with git access can regenerate all certificates

**Remediation**: Remove the CA key from the bind mount. Only the CA cert (`root-ca.pem`) should be shared. The CA key should stay on the host.

## Layer 2: Traefik HTTPS (External)

### Scope

External access via HTTPS. Traefik terminates TLS and routes to backend containers.

### Configuration

| Setting | Value |
|---|---|
| TLS Resolver | `TLS_CERTRESOLVER` from `env/.env` |
| Self-signed | `TLS_CERTRESOLVER=""` (empty) → Traefik self-signed fallback |
| Let's Encrypt | `TLS_CERTRESOLVER=letsencrypt` → requires `ACME_EMAIL` + public domain |
| HTTP Port | `TRAEFIK_HTTP_PORT` (default 8000, adjustable) |
| HTTPS Port | `TRAEFIK_HTTPS_PORT` (default 8443) |

### ACME / Let's Encrypt

1. Set `TLS_CERTRESOLVER=letsencrypt` in `env/.env`
2. Set `ACME_EMAIL=your@email.com` in `traefik/.env`
3. Set `DASHBOARDS_DOMAIN=real-public-domain.com` in `env/.env`
4. Ensure port 80 is reachable from the internet
5. Traefik will request and store certificates in `traefik/acme.json`

⚠️ **Note**: The OPAC and Staff routes do NOT have explicit TLS labels in the docker-compose.yml. Only dashboards has both `web` and `websecure` router labels. Koha OPAC/Staff may need additional labels for full HTTPS coverage.

### OpenSearch CA Cert for Koha

When `OPENSEARCH_CA_CERT` is set:
1. The CA cert is bind-mounted into the koha container
2. Koha connects to OpenSearch over HTTPS instead of HTTP
3. Without this, Koha → OpenSearch is plaintext (even when OS has TLS)

**Security gap**: The CA cert is set to `/dev/null` by default. Koha connects to OpenSearch in plaintext unless explicitly configured.

## Known Security Issues (from ISSUES.md)

### Critical

| # | Issue | Details | Remediation |
|---|---|---|---|
| 1 | `cap_add: ALL` | Koha container has every Linux capability | Reduce to `SYS_NICE` + `IPC_LOCK` only |
| 2 | Plaintext secrets | Passwords in `env/defaults.env` | Move to gitignored `env/.env` |
| 3 | TLS verification disabled | Default: no client cert, no HTTPS to OS | Set `OPENSEARCH_CA_CERT`, use HTTPS |
| 4 | CA private key exposed | `root-ca-key.pem` in all containers | Remove from bind mount |

### High

| # | Issue | Details |
|---|---|---|
| 5 | Missing external networks | `knonikl`, `opensearch-36_osearch` not created by compose |
| 6 | UID collision | `userdel -r ubuntu` in Dockerfile is a hack |
| 7 | No Koha healthcheck | No Docker healthcheck on koha container |

### Medium / Low

- Mixed image/build semantics
- Dashboard port drift (label hard-coded vs env var)
- CRLF normalization redundancy
- IPv6 risk (some services have no IPv6 controls)
- `latest` tag on memcached image

## Capability Analysis

The `cap_add: ALL` on the koha container grants:
- `CAP_NET_ADMIN` — network configuration
- `CAP_SYS_ADMIN` — mount, namespaces, many kernel operations
- `CAP_DAC_OVERRIDE` — bypass file permissions
- `CAP_SYS_PTRACE` — debug any process
- ... and 38+ other capabilities

### Recommended Minimum

| Capability | Why |
|---|---|
| `SYS_NICE` | Set process priority, CPU affinity |
| `IPC_LOCK` | Lock memory (needed by some services) |

The full list of capabilities can be reviewed with:
```bash
docker inspect koha-docker-koha-1 --format '{{.HostConfig.CapAdd}}'
```

## Password Sync Mechanism

`stack.sh` maintains password consistency between `env/.env` and `OpenSearch-3.6/.env`:

```bash
sync_koha_opensearch_credentials() {
  # Source of truth: OpenSearch-3.6/.env
  # Target: env/.env (ELASTIC_OPTIONS userinfo)

  OPENSEARCH_INITIAL_ADMIN_PASSWORD=$(python3 -c "import re; ...")
  
  # Sync Koha DB password
  KOHA_DB_ROOT_PASSWORD=$(grep -E '^KOHA_DB_ROOT_PASSWORD=' OpenSearch-3.6/.env | cut -d= -f2)
  export KOHA_DB_ROOT_PASSWORD

  # Sync OpenSearch admin password
  KOHA_DB_PASSWORD=$(grep -E '^OPENSEARCH_INITIAL_ADMIN_PASSWORD=' OpenSearch-3.6/.env | cut -d= -f2)
  export KOHA_DB_PASSWORD

  # Sync ELASTIC_OPTIONS with Python regex
  ELASTIC_OPTIONS=$(python3 -c "import re, os; ...")
}
```

The Python regex replaces the `<password>` element in ELASTIC_OPTIONS with the current password. This ensures Koha can authenticate to OpenSearch.
